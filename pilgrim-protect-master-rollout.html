How Data Flows
Step 1 Admin populates schools into MongoDB (name, district, GPS coordinates, room count, student count) Step 2 Field workers open Expo app → log in → see their assigned schools Step 3 Worker arrives at school → opens spray report form → fills date, rooms sprayed, takes before/after photos → submits Step 4 Report hits Express API → photos upload to Cloudinary → report saves to MongoDB → school's lastSprayDate updates Step 5 Donor visits website → Mapbox map loads GeoJSON from /api/stats/map → sees all schools with real spray data Step 6 Donor clicks school marker → popup shows name, spray history, worker photos → clicks "Sponsor This School" Step 7 Stripe Checkout processes payment → webhook fires → Donation record created → Donor linked to school → Brevo sends receipt email Step 8 Donor returns to portal → sees their sponsored schools → gets email updates when workers submit new spray reports

Dependencies — What Blocks What
Backend Setup → blocks Everything. Nothing works without the API and database. Seed Data → blocks Worker App (needs schools to select) + Map (needs coordinates to plot) Worker App → blocks Live spray data on the map. Can build map UI in parallel, but real data needs workers submitting. Map + Website → blocks Donor features. Stripe "Sponsor a School" needs school profiles to exist first. Stripe + Donors → blocks Brevo email triggers (donation receipts, sponsor update emails). Can run in parallel → Worker App (Phase 3) + Seed Data (Phase 2) can overlap. Map UI buildout can start while workers test the app.

1

Backend Foundation

Express server · MongoDB schema · Auth · Cloudinary
Week 1–2 ▾
Server Setup
Initialize Express project with folder structure (routes/, models/, middleware/, config/)
Set up environment variables (.env) — MongoDB URI, JWT secret, Cloudinary keys
Configure CORS (allow web + mobile origins)
Add rate limiting (express-rate-limit)
Error handling middleware
Database & Models
Connect to MongoDB Atlas (free M0 cluster)
Create Mongoose models: School, SprayReport, Worker, Donor, Donation
Add 2dsphere geospatial index on School.location
Set up model validations and required fields
Create relationship refs between collections
Authentication
Password hashing with bcrypt
JWT token generation and verification
Auth middleware — protect routes, extract user from token
Role-based guards: admin, worker, donor
POST /api/auth/register, POST /api/auth/login, GET /api/auth/me
Core Routes + Uploads
CRUD: GET/POST/PUT /api/schools
POST /api/spray-reports (worker submits)
GET /api/spray-reports (filterable by school, worker, date)
POST /api/upload/image — Cloudinary integration with multer
Test all endpoints with Postman / Thunder Client
✓ Deliverable: Working REST API. You can create schools, register workers, submit spray reports with photos, and authenticate — all via API calls. No UI needed yet.
2

Populate Real Data

School records · GPS coordinates · Seed script · MAP malaria data
Week 3 ▾
School Data Collection
Gather real school names and districts from Pilgrim Africa contacts
Look up GPS coordinates for each school (Google Maps / OpenStreetMap)
Collect room counts and student enrollment where available
Organize into structured JSON file (schools.json)
Seed Script & Context Data
Build seed.js script that loads schools.json into MongoDB
Create admin user and test worker accounts in seed
Pull Malaria Atlas Project data — district-level parasite rates
Store MAP data in MalariaData collection as context layer
Build GET /api/stats/map endpoint — returns GeoJSON FeatureCollection
✓ Deliverable: Database populated with real schools and real coordinates. /api/stats/map returns GeoJSON ready for Mapbox. Every school is a plottable point.

Set up navigation — React Navigation (stack navigator)

Core Screens (4 total)

Spray Report Form — date picker, room count input, notes field

My Reports — list of past submissions from /api/spray-reports/mine
Offline & GPS

Offline queue — detect no internet → save report to AsyncStorage
Background sync — when connectivity returns, push queued reports
Visual indicator: "1 report pending upload"
Testing
Test on Expo Go (iOS + Android)
Test photo uploads → verify they appear in Cloudinary
Test offline → airplane mode → submit → reconnect → verify sync

Donor dashboard: sponsored schools, donation history, total impact
Success/thank you page after donation with social share links
✓ Deliverable: Donors can sponsor a school directly from the map or school profile. Payments are processed, receipts are generated, and donors have a portal to track their impact.
